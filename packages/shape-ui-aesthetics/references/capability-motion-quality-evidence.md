# Motion Quality Evidence

**Stable ID:** `motion-quality-evidence`  
**Class:** Core  
**Primary domain:** Coherence, Critique & Evidence

## Observable trigger

Route here when an already named motion purpose and concrete hypothesis or implemented artifact needs a defensible verdict across utility, elegance, accessibility, and real performance.

## Non-trigger and near miss

Do not route here when no temporal job has been named; return `reframe` and signal Motion, Rhythm & Causality so purpose is established first. Do not route here only to select CSS versus JavaScript or tune one known parameter.

## Decision question

Given the accepted temporal job, does this motion hypothesis or artifact deliver useful explanation, product-fitting character, accessibility, and real performance—and which evidence would improve or reject it?

## Interpretation method

Use three gates. Utility asks whether motion improves understanding, feedback, control, continuity, or attention. Elegance asks whether its rhythm and character fit the product without delay or distraction. Performance asks whether target devices, real content, repetition, and concurrent work preserve response. Utility is admission; elegance cannot offset usability loss. Tune uncertain properties through single-variable experiments.

## Executable design procedure

1. Name the motion, user event, purpose, and observable success.
2. Evaluate utility with task evidence and a no-motion comparison.
3. Evaluate elegance against thesis, repetition, interruption, and attention hierarchy.
4. Profile performance under target device, content, concurrency, power, and input conditions.
5. State specific failure modes and controlled questions for duration, easing, distance, stagger, or representation.
6. Run one-variable comparisons and preserve raw observations.
7. Keep, revise, degrade, or remove motion; protect state and completion feedback first.

## Boundaries and failure modes

- Do not use motion to disguise a boring or broken flow.
- Do not let elegance compensate for missing utility or accessibility.
- Do not infer performance from historical browser assumptions or a developer machine.
- Include reduced motion, keyboard focus, screen reader independence, and cognitive load.
- Remove decorative layers before task feedback when resources fail.

## Attached Reference Knowledge

Quality is a conjunction of useful explanation, product-fitting character, and real execution. Prototype fidelity should match the question: low fidelity for timing and sequence hypotheses, high fidelity for material, performance, and interaction truth.

## Required Capability Contribution

## Runtime Self-Critique

Ask whether the verdict would survive without seeing the implementation technique, whether testing matches real load, and whether the retained motion serves the task before taste.
