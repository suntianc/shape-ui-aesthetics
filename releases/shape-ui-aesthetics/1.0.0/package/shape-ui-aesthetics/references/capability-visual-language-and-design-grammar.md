# Visual Language & Design Grammar

**Stable ID:** `visual-language-and-design-grammar`  
**Class:** Core  
**Primary domain:** Coherence, Critique & Evidence

## Observable trigger

Route here when local aesthetic decisions no longer read as one system across components, states, surfaces, or platforms; similar roles diverge without reason; or implementation values exist without shared semantic relationships.

## Non-trigger and near miss

Do not route here for a simple inventory of existing styles, for a single isolated implementation bug, or when meaningful product contexts require different expressions that already share a governing relationship. Consistency is not a reason to erase necessary contrast or local identity.

## Decision question

What compact set of semantic roles, generative relationships, constraints, and purposeful exceptions lets the product remain recognizably itself across content, state, surface, and implementation?

## Interpretation method

Treat visual language as grammar rather than a palette of parts. Decompose the artifact into color, typography, spacing, composition, shape, iconography, imagery, material/depth, motion, and sound where applicable. For each dimension, define semantic roles, relationships, constraints, accessibility obligations, and conditions for exception.

Represent repeated decisions as semantic tokens or equivalent implementation-visible rules without reducing the grammar to raw values. Let platforms implement the same intent differently when their medium demands it. Derive the grammar from the accepted protagonist, ritual, thesis, and Signature Move; do not let a design system replace authored direction.

## Executable design procedure

1. Inventory repeated and contradictory expressions across real components, content, states, viewports, and surfaces.
2. Trace each important expression to a product role, interaction state, hierarchy function, or aesthetic thesis consequence.
3. Define semantic roles and governing relationships for color, type, spacing, composition, shape, imagery, depth, and motion as applicable.
4. Encode constraints, accessibility requirements, and permitted ranges before raw implementation values.
5. Define purposeful exception rules and prevent exceptions from becoming parallel grammars.
6. Map the grammar to platform or code primitives while preserving intent rather than copying implementation details.
7. Test long content, cardinality, empty/loading/error/success, focus/disabled/selected states, responsive regimes, localization, and reduced motion.
8. Remove orphan values, duplicate roles, and rules unsupported by thesis or task evidence.

## Boundaries and failure modes

- Do not standardize away the protagonist, Signature Move, meaningful hierarchy, cultural distinction, or necessary state contrast.
- Do not treat raw colors, font sizes, or spacing values as a complete semantic grammar.
- Avoid token proliferation, synonym roles, and platform copies that silently diverge in intent.
- Do not claim coherence from documentation when the rendered artifact contradicts it.
- Signal the originating domain when grammar drift reveals an unresolved spatial, information, color, motion, interaction, medium, human-fit, or meaning decision.

## Attached Reference Knowledge

A shared vision and principles govern why the system exists; a pattern library captures repeatable expressions; testing protects state and implementation truth; performance constrains fonts, assets, and motion. These supports remain subordinate to the product-specific grammar. Cross-platform consistency means shared semantic intent, not pixel identity.

## Runtime Self-Critique

Ask whether a new component can be derived without inventing a parallel style, whether exceptions remain meaningful, whether the rendered artifact matches semantic roles, and whether coherence has flattened the product's most authored move.
