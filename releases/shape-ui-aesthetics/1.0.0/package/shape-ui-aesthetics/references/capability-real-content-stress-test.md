# Real-Content Stress Test

**Stable ID:** `real-content-stress-test`  
**Class:** Core  
**Primary domain:** Coherence, Critique & Evidence

## Observable trigger

Route here when a direction has only been judged with ideal copy, imagery, cardinality, state, permission, locale, or viewport conditions.

## Non-trigger and near miss

Do not route here before a testable component or artifact exists, or when the known defect already reproduces with real content and requires a targeted correction rather than a broader matrix.

## Decision question

Which high-probability real content, state, and environment variants break the direction, and what system-level rule must change rather than patching each example?

## Interpretation method

Use pages and flows as test surfaces for the design system. Build a prioritized matrix of actual names, lengths, missing media, permissions, densities, locales, states, devices, and failures. Feed variants through components and full compositions. Treat repeated breakage as evidence against the governing rule, not as isolated content inconvenience.

## Executable design procedure

1. Identify the artifact's real content sources, cardinalities, roles, and highest-cost failures.
2. Build a bounded matrix covering long/short, empty/full, missing, loading, stale, error, permission, localization, and consequential viewports.
3. Exercise components, compositions, interactions, and assistive states with the matrix.
4. Record visible, behavioral, semantic, responsive, and performance breakage.
5. Prioritize by probability and consequence; avoid infinite edge enumeration.
6. Correct tokens, grammar, component, composition, or routing rules at the earliest shared cause.
7. Re-run the matrix and retain representative regression evidence.

## Boundaries and failure modes

- Do not approve static ideal comps as evidence of production resilience.
- Do not replace real data with convenient demo content to pass the test.
- Do not expand a bounded Refine task beyond its Change Boundary; report external causes separately.
- Add current automated visual, accessibility, token, and component testing where relevant.
- Preserve privacy and use representative safe fixtures when real data cannot be exposed.

## Attached Reference Knowledge

Templates show intended structure; real pages reveal whether it holds. Early browser prototypes and inventories expose content, state, and platform truth sooner than polished static images.

## Required Capability Contribution

Return exactly the canonical seven fields from `routing-and-reconciliation.md`: `decision`, `evidence`, `recommendation`, `consequence`, `boundary`, `validation`, and `onward`. Do not add a mini-report.

Populate those fields with the decision key; prioritized content/state matrix; breakage evidence; earliest shared causes; system-level corrections; remaining bounded risks; regression plan; and the onward state.

## Runtime Self-Critique

Ask whether the matrix reflects actual probability and consequence, whether fixes generalize, and whether ideal content is still hiding any required state or role.
